# 実装の選択と根拠

## プラットフォーム: Streamlit
- **理由**: チャットUI (`st.chat_message`, `st.chat_input`) が標準搭載されており、チャットベースの経費入力に最適。テーブル表示 (`st.dataframe`) やCSVダウンロード (`st.download_button`) も組み込み機能で対応可能。デプロイも Streamlit Community Cloud で無料で可能。
- **却下案**: Gradio はチャットUIのカスタマイズ性が低い。Flask/FastAPI は UI を別途構築する必要がある。

## データベース: SQLite (WALモード)
- **理由**: 個人利用・単一プロセスのため十分。ファイル1つで完結し、バックアップも容易。WALモードにより読み書きの並行性が向上。
- **却下案**: PostgreSQL はオーバースペック。JSON/CSV は検索・集計が不便。

## DBスキーマ設計
- **threads テーブル**: `created_at` と `updated_at` を分離。リネームしても作成日時は保持され、ソートに利用可能。
- **expenses テーブル**: `amount` は INTEGER（日本円のみのため小数不要）。外部キーに ON DELETE CASCADE を設定し、スレッド削除時に関連データも自動削除。
- **messages テーブル**: チャット履歴を保持し、AI に文脈を渡すために使用。role は 'user' と 'assistant' に限定。

## AI連携方式
- **JSON埋め込み方式**: AIの応答テキスト内にJSON形式で経費データを埋め込ませ、正規表現で抽出する方式を採用。
- **理由**: Function Calling に依存せず Claude/OpenAI 両対応が容易。プロンプトで出力形式を指定するだけで済む。
- **却下案**: Function Calling は provider ごとに実装が異なり、切り替え機能との相性が悪い。

## AIモデル選択
- **Claude**: claude-haiku-4-5-20251001 — 費目分類程度のタスクには十分な性能で、コストが低い。
- **OpenAI**: gpt-4o-mini — 同様にコスト効率が良い。
- **理由**: 経費の分類・抽出は比較的単純なタスクのため、最上位モデルは不要。

## AI への文脈渡し方式
- **現在の経費一覧をユーザーメッセージに付加**: チャット履歴全体＋現在の経費一覧をAIに渡す。
- **理由**: AIが「集計して」「合計は？」などの質問に正確に答えられるようにするため。システムプロンプトに動的データを入れるより、ユーザーメッセージに付加する方がシンプル。

## UI構成
- **サイドバー**: スレッド一覧、AI切り替え、新規作成。常に表示。
- **メインエリア**: タブで「チャット」と「集計・出力」を切り替え。
- **理由**: チャットと集計を同時に見る必要性は低く、タブで分離した方がすっきりする。

## 表示からJSONを除去
- **AI応答からJSONブロックを除去して表示**: ユーザーにはJSONを見せず、代わりに「記録しました: ...」と整形表示。
- **理由**: JSONはユーザーにとってノイズ。記録された内容を人間が読みやすい形で確認できる方が良い。

## 通貨
- **日本円固定、INTEGER型**: 小数点以下は不要。表示時に `¥{:,}` でフォーマット。
